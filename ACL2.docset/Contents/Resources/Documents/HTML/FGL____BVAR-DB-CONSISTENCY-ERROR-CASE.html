Case macro for the different kinds of bvar-db-consistency-error structures.

This is an ACL2::fty sum-type case macro, typically introduced by defflexsum or fty::deftagsum. It allows you to safely check the type of a bvar-db-consistency-error structure, or to split into cases based on its type.

Short Form

In its short form, bvar-db-consistency-error-case allows you to safely check the type of a bvar-db-consistency-error structure. For example:

(bvar-db-consistency-error-case x :eval-error)

is essentially just a safer alternative to writing:

(equal (bvar-db-consistency-error-kind x) :eval-error)

Why is using bvar-db-consistency-error-case safer? When we directly inspect the kind with equal, there is no static checking being done to ensure that, e.g., :eval-error is a valid kind of bvar-db-consistency-error structure. That means there is nothing to save you if, later, you change the kind keyword for this type from :eval-error to something else. It also means you get no help if you just make a typo when writing the :eval-error symbol. Over the course of developing VL, we found that such issues were very frequent sources of errors!

Long Form

In its longer form, bvar-db-consistency-error-case allows you to split into cases based on the kind of structure you are looking at. A typical example would be:

(bvar-db-consistency-error-case x
  :eval-error ...
  :inconsistency ...)

It is also possible to consolidate ``uninteresting'' cases using :otherwise.

For convenience, the case macro automatically binds the fields of x for you, as appropriate for each case. That is, in the :eval-error case, you can use defprod-style foo.bar style accessors for x without having to explicitly add a eval-error b* binder.
